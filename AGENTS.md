# InkOS Agent Constraints

本文件为仓库级硬约束。除系统级指令外，进入本仓库工作的 agent 必须遵守本文件与 [WORKFLOW.md](/D:/workspaces/cli/inkos/WORKFLOW.md)。

## 适用范围

- 适用于 `D:\workspaces\cli\inkos` 整个仓库。
- 适用于产品、设计、原型、前端、后端、联调、验证、交付全链路工作。
- 当仓库处于实现阶段时，本文件中的 loop 开发要求为强约束。

## 当前阶段判定

- 当前仓库已从文档驱动阶段进入实现阶段。
- 进入实现阶段后，不允许停留在“只做计划、只做分析、只给建议、只改原型、不落代码”的状态。
- 前端与后端必须并行推进，并持续对齐需求文档、设计文档、原型和真实实现。

## 全量 Loop 开发硬约束

- agent 必须以“持续闭环开发”方式工作，直到当前目标真正落地，而不是停在中间产物。
- loop 必须覆盖：需求/设计对齐、任务拆解、代码实现、mock 补齐、联调、验证、修复、复审、再次修复、交付记录。
- 除非出现真实阻塞，agent 不得以“等待确认”“先给方案”“先总结现状”作为默认停点。
- 如果存在可继续推进的路径，agent 必须继续推进，不得主动停止。
- 每轮 loop 都必须产出可验证增量，禁止空转式反复讨论。

## Plane 为主链硬约束

- Plane 是本仓库开发执行主链，`plane-cli` 是默认入口。
- 开始任何中大型任务前，必须先确认对应 module、issue、state、workpad。
- 开发过程中必须持续更新 Plane 记录，至少包括：
  - 当前范围
  - 当前实施步骤
  - 已完成改动
  - 验证结果
  - 剩余风险
  - 下一步动作
- 开发、评审、交付必须通过 Plane 状态流推进，不允许脱离 Plane 单独失控开发。
- 如果 `plane-cli` 本身存在 bug、配置错误、角色映射错误或写入异常，必须先修复或绕过，并继续主任务，不得以此停工。

## 前后端并行落地硬约束

- 真实产品实现必须覆盖前端和后端，不能只完成其中一侧。
- 前端必须采用 `assistant-ui` runtime 与 Tool UI 作为主交互承载，不允许退化成普通聊天文本或与文档脱节的自定义协议壳。
- 前端页面、状态、交互、错误态、空态、加载态、移动端适配都必须可运行。
- 后端接口、事件流、持久化、mock 数据、状态转换、错误返回都必须可运行。
- 前后端协议必须以文档为准并在实现中保持一致；发现冲突时必须先修正文档或实现，再继续推进。
- 原型仅作为验证参考，不能替代正式实现。
- 正式 UI 必须以 `docs/prototypes/` 下的 v3 原型作为参照样板，优先继承其主题变量、页面信息结构、导航层级和 Tool UI 形态，不得另起一套脱节视觉体系。

## 模块化与文件规模硬约束

- 前后端都必须模块化实现，优先按职责拆分为 schema、store、service、transport、ui、hooks、components 等层，不得把多种职责长期堆在单文件中。
- 单文件应尽量控制在 `500-600` 行以内；接近该阈值时应优先继续拆分，而不是继续追加逻辑。
- 解耦优先于短期拼接：类型定义、协议 schema、数据访问、业务编排、UI 组件、样式和测试应分层放置。
- 若因当前阶段临时实现导致文件偏大，必须在后续 loop 中优先拆分回合理规模。

## Mock 与功能有效性硬约束

- 在真实能力尚未全部接通前，必须使用完整 mock 保证功能链路可演示、可交互、可验证。
- mock 不得只做静态截图式占位，必须具备状态变化、用户反馈和关键路径闭环。
- 一切用户可见功能都应至少具备可运行的 mock 或真实实现之一。
- 当真实后端能力补齐后，必须逐步替换临时 mock，避免长期双轨失真。

## 审查与复审硬约束

- 所有重要改动都必须经过至少一轮实现后复审。
- 如任务要求依赖 Gemini、UX 专家或其他评审视角，agent 必须将其意见转化为实际修正，而不是只记录结论。
- 复审发现的问题必须继续进入下一轮 loop，直到没有高优先级阻塞问题为止。
- 不得以“已初步可用”替代“已闭环验收”。

## 验证硬约束

- 每轮开发结束前必须执行与改动相匹配的验证。
- 验证至少覆盖：
  - 静态检查
  - 关键路径手动验证
  - 前后端联调验证
  - 主要异常路径验证
- 若因环境限制无法完成某项验证，必须明确记录缺口，并继续完成其他可执行验证。
- 不允许在未验证的情况下把任务宣告为完成。

## Windows 本地服务硬约束

- 当前用户环境是 Windows，本地长期运行的 `web` / `server` 服务不得再通过会弹出前台 `cmd` 窗口的方式启动。
- 默认使用仓库脚本：
  - `pnpm dev:hidden:stack`
  - `pnpm dev:hidden:server`
  - `pnpm dev:hidden:web`
  - `pnpm dev:hidden:stop`
- 隐藏启动日志统一写入 `output/dev/`，pid 也写入同目录，停止时优先走 pid 文件和目标端口清理。
- 如果隐藏启动方案失效，必须先修复该方案，再继续联调或浏览器验证。

## 浏览器联调硬约束

- 进行 Playwright 真浏览器联调时，默认复用单一 session id，不得在同一轮任务里频繁新开多个浏览器线程。
- 当前仓库默认复用的 session id 为 `inkos-loop`。
- 只有当既有 session 明确损坏、卡死或 CLI daemon 无法恢复时，才允许切换到新的 session id；切换后应继续长期复用新的 id，而不是再次散开。

## LLM 测试提供方硬约束

- 当需要进行真实 LLM 联调、端到端生成验证或 `packages/core` 能力回归时，默认优先使用 BigModel 新接口，不再默认使用历史上反复失效的旧 `glm` 配置。
- 默认测试基线如下：
  - Base URL: `https://open.bigmodel.cn/api/coding/paas/v4`
  - 首选模型: `glm-5`
  - 兼容回退模型: `glm-4.7`
  - 迁移参考: `https://docs.bigmodel.cn/cn/guide/start/migrate-to-glm-new`
- 密钥不得硬编码进仓库文件、源码或可提交配置中；必须通过本地安全环境变量或未跟踪本地配置注入。
- 如无特殊说明，联调时读取：
  - `INKOS_LLM_PROVIDER=openai`
  - `INKOS_LLM_BASE_URL=https://open.bigmodel.cn/api/coding/paas/v4`
  - `INKOS_LLM_MODEL=glm-5`
  - `INKOS_LLM_API_KEY` 从本地安全环境注入
- 若 `glm-5` 存在兼容性问题，可回退 `glm-4.7`，但必须记录回退原因与现象。

## 停止条件硬约束

- 只有满足以下条件，当前 loop 才能视为完成：
  - 需求和设计已对齐到当前实现
  - 前端和后端关键链路可运行
  - 必要 mock 已补齐
  - 关键验证已通过
  - Plane 记录已更新
  - review 问题已收敛到可接受范围
- 若上述条件未满足，agent 默认继续下一轮 loop。

## 执行细则

- 优先修复真正阻塞后续开发的问题。
- 优先完成端到端最小闭环，再逐步扩展覆盖面。
- 不得为了追求局部完美而阻塞整体主链闭环。
- 不得以总结、汇报、计划文档替代代码、验证和联调产出。
- 若发现仓库规则缺失，应先补规则，再继续执行。

## 与 WORKFLOW 的关系

- [WORKFLOW.md](/D:/workspaces/cli/inkos/WORKFLOW.md) 定义仓库整体流程与 Plane 规则。
- 本文件补充“实现阶段必须执行全量 loop 开发”的硬约束。
- 当 issue 已进入实现或联调阶段时，优先执行本文件的持续闭环开发要求。

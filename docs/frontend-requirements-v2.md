# InkOS 前端需求文档 v2

## 1. 文档目标与阶段边界

本文档定义 InkOS Web 前端 v2 在当前阶段的产品需求基线。

当前阶段不是正式开发阶段，而是 **文档冻结前阶段**。目标是先把需求边界、关键场景、完成面职责和原型验证范围收敛清楚，再进入 HTML 原型验证，最后才进入开发拆解。

当前阶段的唯一目标：

- 收敛需求文档
- 收敛设计文档
- 为后续原型验证和开发拆解提供稳定基线

当前阶段不做：

- 不直接进入正式前端实现
- 不提前批量拆开发 issue
- 不把 HTML 原型当成正式实现代码
- 不把 Plane Pages 当作唯一文档源

进入下一阶段的条件：

1. 本文档的核心场景、页面职责、风险边界、交互承载矩阵已稳定
2. 设计文档已冻结最小运行时协议和系统边界
3. 才允许进入 HTML 原型验证阶段

## 2. 文档治理与 Plane 协作方式

## 3. 产品定位与非目标

InkOS Web 是一个 **本机单用户小说生产控制台**。

产品定位：

- 以总编 Agent 作为唯一直接面向用户的 AI 角色
- 把建书、写作、审计、修订、导入、调度、素材生成收敛为统一工作流
- 让用户在聊天页完成大多数基础任务，而不是在终端、文档和多个后台页面之间切换
- 让所有自动执行都可见、可取消、可追踪、可复盘

非目标：

- 不做多人协作
- 不做登录系统
- 不做公网 SaaS
- 不直接调用 CLI 二进制执行任务
- 不要求移动端等价承载全部生产操作
- 不把内部运行时 skill 暴露为普通用户必须理解的主概念

## 5. 用户角色与最高优先级任务

### 5.1 独立作者

最重要的任务：

- 新建一本书
- 继续写下一章
- 审计失败后快速理解问题
- 生成角色等素材并保存

### 5.2 多书运营者

最重要的任务：

- 看多本书进度与风险
- 看当前自动化状态
- 快速切换上下文

### 5.3 深度调优用户

最重要的任务：

- 调整文风与导入资料
- 查看 materials 是否进入写作上下文
- 在高级设置中查看内部能力状态

## 6. v2 首批必须闭环的 4 个场景

当前阶段先把以下 4 个场景写死，其他场景属于兼容后续扩展：

### 6.1 场景 A：建书

用户目标：

- 从无到有建立一本书的基础配置

预期闭环：

- 用户输入意图
- 总编收集缺失信息
- 生成建书计划
- 调用核心能力完成创建
- 返回建书结果，并提供进入书籍页的明确入口

### 6.2 场景 B：继续写下一章

用户目标：

- 在当前书上下文中推进下一章

预期闭环：

- 用户发起写章请求
- 总编生成计划
- 执行写作、审计、修订链路
- 展示轨迹与结果
- 用户决定是否继续分析或跳转专页

### 6.3 场景 C：生成角色并保存

用户目标：

- 通过对话生成角色候选并保存到素材库

预期闭环：

- 聊天内表单补参数
- 返回结构化表格或卡片
- 用户执行 `应用保存 / 重写生成 / 编辑表单 / 取消`
- 系统把结果写入 materials

### 6.4 场景 D：诊断审计失败

用户目标：

- 理解某章为什么没通过审计，并得到下一步建议

预期闭环：

- 从章节页或 `/chief` 发起分析
- 总编读取审计结果和上下文
- 返回结构化问题说明
- 必要时提供进入修订流或章节专页的明确入口

## 7. 交互承载矩阵

这是 v2 的核心产品约束。

### 7.0 场景与完成面映射

| 场景 | 默认完成面 | 说明 |
|---|---|---|
| A：建书 | `/chief` | 多步补信息与建书结果确认 |
| B：继续写下一章 | `/chief` | 计划、执行轨迹与结果都在主工作台完成 |
| C：生成角色并保存 | chat 内闭环，必要时跳 `/chief` | 会话界面默认完成；复杂候选比较时升级到 `/chief` |
| D：诊断审计失败 | `/chief`，必要时跳章节页 | 结构化说明在 `/chief` 完成；长文本阅读在章节页完成 |

### 7.1 chat 内闭环

必须能在聊天内完成：

- 参数补充
- 计划展示
- 角色类 materials 生成
- 结果确认与保存
- 轻量诊断说明

这里的 “chat 内闭环” 指 **会话界面中的事务闭环**，不是指“必须在右下快速 modal 中闭环”。

### 7.2 `/chief` 完成

必须在 `/chief` 完成：

- 多步事务
- 审批动作
- 生成候选的比较与确认
- 长时间执行的任务
- 需要持续跟踪的线程

### 7.3 专页完成

必须在专页完成：

- 大文本阅读
- 长 diff 浏览
- truth files 浏览
- materials 列表管理
- 调度监控

### 7.4 modal 承载范围

右下快速入口只适合：

- 快速问答
- 页面内轻量查询
- 单步轻操作

以下情况必须升级到 `/chief`：

- 需要审批
- 需要展示复杂的、多候选结构化对比或长文本详情
- 工具调用超过 1 次
- 预计执行超过 30 秒
- 需要跨页面理解上下文

补充说明：

- 轻量的单卡结构化结果允许留在 chat 或快速 modal 内完成，只要仍是单步事务
- 涉及跳转到书籍页、章节页、修订流的场景，界面必须提供明确可点击入口，不能只靠 assistant 文本暗示

## 8. 总编 Agent 的产品职责与边界

### 8.1 产品职责

总编 Agent 是系统的统一调度者。

职责：

- 理解用户目标
- 绑定当前上下文
- 决定是否缺参数
- 制定计划
- 调度底层能力
- 汇总结果

### 8.2 用户可感知能力

用户看到的是：

- 总编会提问补信息
- 总编会展示计划
- 总编会解释系统做了什么
- 总编会要求确认高风险操作
- 总编会输出结构化结果

### 8.3 内部能力

系统内部会发生：

- worker 路由
- skill 解析
- core 调用
- 事件记录
- materials 写入

这些不作为普通用户主交互的一部分暴露。

### 8.4 自动执行边界

默认可自动执行：

- 只读分析
- 轻量查询
- 生成草案
- 可撤销的 materials 草案保存

必须暂停并等待用户：

- 章节重写
- 回滚
- 导入覆盖
- 修改全局模型或调度配置

必须只分析不执行：

- 用户意图不明确
- 当前上下文不足以安全写入
- 预计会改变大量项目事实但用户未确认

### 8.5 拒绝与升级条件

总编必须拒绝或转人工的情况：

- skill 缺失且无安全降级路径
- 当前动作会破坏项目事实但用户未确认
- 依赖信息缺失且无法合理推断

## 9. 风险分级与审批规则

风险等级：

- `low`：只读分析、生成草案、不改写项目事实
- `medium`：写入可撤销内容、更新 materials
- `high`：改写章节、truth files、导入覆盖历史、修改全局策略

审批矩阵：

| 动作 | 风险 | 是否审批 |
|---|---|---|
| 读取状态 / 分析问题 | low | 不审批 |
| 生成素材草案 | low | 不审批 |
| 保存素材到 materials | medium | 不走 Agent 审批，但必须可撤销 |
| 丢弃未应用草案 | low | 需要 UI 二次确认 |
| 重写章节 | high | 必须审批 |
| 回滚快照 | high | 必须审批 |
| 导入章节并覆盖索引 | high | 必须审批 |
| 修改全局模型或调度配置 | high | 必须审批 |

## 10. 信息架构与首版页面范围

### 10.1 一级页面职责

- `/`：仪表盘
- `/chief`：总编工作台
- `/books`：书籍列表
- `/books/:bookId`：书籍总览
- `/books/:bookId/chapters/:chapterNo`：章节工作台
- `/books/:bookId/truth`：truth files 中心
- `/books/:bookId/materials`：素材中心
- `/automation`：调度中心
- `/settings`：设置中心

### 10.2 首版必须重点收敛的页面

当前阶段优先收敛：

- `/chief`
- `/books/:bookId/materials`
- 章节分析的完成面边界

当前阶段只定义职责、不做完整细节的页面：

- `/truth`
- `/automation`
- `/settings`

## 11. Materials 产品需求

### 11.1 首批冻结范围

v2 当前阶段只冻结 3 类高价值 materials：

- `character`
- `faction`
- `location`

以下类型列入后续兼容扩展，不在当前阶段冻结完整交互：

- `world-rule`
- `scene-card`
- `hook`
- `角色关系`
- `章纲候选`

### 11.2 核心交互

materials 的基础事务流：

1. 补参数
2. 生成候选
3. 显示结构化结果
4. `应用保存 / 重写生成 / 编辑表单 / 取消`

### 11.3 草案状态

当前阶段先冻结以下状态语义：

- `draft`
- `applied`
- `discarded`
- `failed`

## 12. assistant-ui 产品要求

assistant-ui 在当前阶段承担：

- 主线程容器
- 快速弹层容器
- Tool UI 容器

必须满足：

- 表单、结果卡、审批卡不能退化为纯文本
- chat 内基础事务必须能闭环
- Tool UI 有明确状态语义
- modal 升级规则稳定

## 13. 统一样式系统产品要求

当前阶段必须先冻结这些视觉规则：

- light / dark 双主题都要存在，且都能覆盖 `/chief` 关键交互
- 页面栅格
- 内容最大阅读宽度
- 聊天气泡与业务卡片的层级关系
- 主按钮 / 次按钮 / 危险按钮等级
- 表格密度
- 卡片密度与留白

当前阶段不冻结：

- 所有页面的最终细部布局
- `/chief` 最终三栏宽度比例
- 工件检查器是否常驻

## 14. Skill 系统的用户可见性与设置要求

### 14.1 用户可见性

skill 默认对普通用户隐藏。

普通用户看到的是：

- 系统能力变化
- 结果是否稳定
- 是否处于稳定轨道或实验轨道

### 14.2 高级设置要求

高级运营者后续可见：

- skill 来源
- 兼容状态
- pin 状态

但这些属于后置实现要求，不是当前阶段原型验证重点。

## 15. 原型验证范围与退出条件

### 15.1 原型在当前阶段的职责

原型只用于验证：

- `/chief` 最小闭环
- modal 升级规则
- Tool UI 的表单卡 / 结果卡 / 审批卡
- light / dark 双主题切换是否稳定
- 统一样式系统是否成立

原型不用于：

- 替代正式需求文档
- 替代正式设计文档
- 提前承诺所有页面细节

### 15.2 原型前置条件

只有当本文档和设计文档都进入可验证状态时，才启动 HTML 原型。

### 15.3 原型退出条件

满足以下条件可视为原型阶段完成：

- `/chief` 最小闭环已验证
- 角色 materials 生成闭环已验证
- modal 升级规则已验证
- light / dark 双主题已验证
- 统一样式核心规则已验证
- 原型反馈已回写文档
